# PLAN.md — パーツ生成方式の現行方針(Gemini移行)

出典: [`amanesf/new2D3D`](https://github.com/amanesf/new2D3D) の
`SUPER_LIVE2D_V3_PLAN.md` §6.5(2026-07-21時点のまとめ)。本ドキュメントは
その内容をこのリポジトリでの実施計画として書き起こしたもの。実装コードは
まだ書いていない — Gemini APIキーが用意され、実際の生成呼び出しが許可され
た時点で着手する。

## 背景: なぜMeinaMix+ControlNet系を離れるのか

`new2D3D`では頭部360度ビューリング(Live2Dでは原理的に不可能な側面・背面
込みの全周表示)の実現手段として、SD1.5系(MeinaMix)+ControlNet /
IP-Adapter / img2imgチェーンを14パターン(①〜⑭)検証した。到達した結論:

- 単発の同一性(髪色・瞳・ヘッドセット・衣装)は、ステップ数とプロンプトの
  トークン管理を詰めれば実用域に達せられる(⑧)
- しかし**複数キーフレームを跨いだ一貫性**(同じ服の形・同じ瞳の色を毎回
  選ぶこと)は保証されない。同一設定で生成しても毎回ゼロから再解釈される
  ため、角度を並べたときに同一キャラの回転に見えない(⑨)
- 生のtxt2img(ControlNet/IP-Adapterなし)+同一seedの方が連続性はやや改善
  するが(⑪)、img2imgチェーンで実ピクセルを引き継ぐと連続性は完璧になる
  一方で回転が止まる(⑫⑬)— 「同一性」と「回転の自由度」がトレードオフ
  になり、SD1.5系の構成では両立しなかった

この構造的な限界を踏まえ、**パーツ生成の主軸をGemini 3.1 Flash Image
Preview(Nano Banana 2)に切り替える**。表情シート(1枚のキャンバスに複数
表情をタイル状に生成)で試したところ、髪色・瞳・ヘッドセット・衣装のいずれ
もMeinaMix系より大幅に安定して揃うことを確認済み(1枚のキャンバス内で生成
することで、モデルの注意機構が同一キャンバス内の情報を自然に参照し合い、
同一性が保たれやすいと推定)。

## 現在の方針

### 再設計の前提(実現性優先)

前回版(§6.5相当)の見積もり(約60回・120枚超)は、可動パーツを角度・
ポーズごとに独立生成する前提だった。しかし`new2D3D`の検証(⑨)では、
**独立生成された複数枚は同一設定でも毎回ゼロから再解釈され、並べたときに
同一キャラの回転に見えない**という致命的な問題が確認されている。一方
Gemini移行の根拠になった表情シート実験は、**1枚のキャンバスに複数タイルを
まとめて生成する**方式だった。この2つの事実から、再設計は次の3原則で
組み立て直す:

1. **生成しない部分を最大化する**: 呼吸・視線・小角度の首振り(±15°)・
   ポニーテールやケープの揺れは、`new2D3D`で既に実証済みの**単一画像への
   メッシュ変形/ソフトマスク**(手続き型、生成不要)で作る。これらは
   生成予算に一切乗せない
2. **生成が要る箇所は「離散状態の差し替え」に限定する**: 表情・viseme・
   手の形・アクションポーズ。単発の破綻は防げても複数枚を跨いだ一貫性は
   保証されない、という⑨の教訓により、**連続的な動き(モーフィング)は
   生成画像同士では作らない**。生成画像はクロスフェードで切り替える離散
   状態としてのみ使う
3. **独立生成をやめ、同一キャンバスへのタイル生成に統一する**: 表情シート
   で実証済みの唯一の高同一性パターンをテンプレート化し、他のカテゴリ
   (viseme・手の形・アクション)にも横展開する。360度ビューリングのような
   独立生成が既に不合格になっている領域は、同一キャンバス方式で
   **作り直す仮説**として扱い、Phase 2の要検証事項に切り出す(本数見積もり
   には含めない)

### 標準テンプレート(2キャラ目再利用の基盤)

- **キャンバス**: 768×1024(縦長)固定。旧解像度(512×704)より引き上げる
  理由は、フル解像度の方が線・識別要素の精度が高いと確認済みのため(⑬)
- **基準ポーズ**: 全身直立、腕は体から離す、脚は軽く開く、髪・ケープは
  体に重ねない(`master_v2_fullbody`のポーズ再設計で確立済みのL0要件。
  重なりがあると抜き・分解自体が破綻する)
- **背景**: 単色固定(キー抜き用)
- **標準ピボット表**: 頭=首の付け根、肩=左右対称位置、股関節=腰位置を
  キャンバス比率(正規化座標)で定義し、次のキャラでもこの比率に合わせて
  ポーズ参照画像を用意すれば、リグ(ボーン位置・ワープメッシュ)を
  流用できるようにする

### パーツ・画像枚数(Phase 1: MVP、生成予算に乗る分のみ)

| カテゴリ | 生成方式 | 呼び出し回数 | 出力タイル数 | 備考 |
|---|---|---|---|---|
| マスター(基準ポーズ) | img2img(既存踏襲) | 1 | 1 | 新規性なし、既存工程そのまま |
| 表情シート | 単一キャンバスにタイル | 1 | 7(neutral+VRM標準6) | 実証済み手法をそのまま踏襲 |
| visemeシート | 単一キャンバスにタイル | 1 | 6(aa/ih/ou/ee/oh/rest) | 表情シートと同型の低リスク手法 |
| 手の形シート | 単一キャンバスにタイル(片手のみ) | 1 | 4(開手/握り/指差し/マイク持ち) | ミラーで反対の手を作る。今回新規のカテゴリなので最初は少量で試す |
| アクションポーズシート | 単一キャンバスにタイル | 1 | 4程度(挨拶/お辞儀/頷き/待機) | 状態間はクロスフェードで繋ぐ(中割りは作らない) |
| 呼吸・視線・首振り±15°・ポニーテール/ケープ揺れ | 生成なし(プロシージャル) | 0 | 0 | 既存メッシュ変形/ソフトマスクを踏襲、追加コストなし |
| **Phase 1 合計** | | **4回** | **約21タイル** | 前回見積もり(60回/120枚超)から大幅圧縮 |

肩肘手首・股膝足首といった「高可動関節の角度継ぎ足し」は、上表からは
**意図的に外した**。大きな動作は表情シート型の「アクションポーズシート」
(挨拶・お辞儀等、離散差し替え)で代替し、関節を角度ごとに切り出して
継ぎ目でつなぐ方式(旧見積もりの主因、約30回)は、継ぎ目の滑らかさが
未検証のままボリュームだけ積んでいたため、Phase 2に格下げする。

### Phase 2(差別化領域、未検証・本数未確定)

- **360度ビューリング**: 角度ごとの独立生成(⑨)は連続性不合格が確定済み。
  再挑戦するなら**同一キャンバスに複数角度をタイル生成する方式**(表情
  シートと同じ技法)を仮説として小規模スパイク(4角度程度)で検証して
  から本数を見積もる。検証前提: Phase 1(正面±15°のプロシージャル疑似3D)
  だけで「スーパーLive2D 1.0」は成立する設計のため、Phase 2が不合格でも
  1.0は成立する
- **関節可動パーツ(角度継ぎ足し)**: 継ぎ目の滑らかさが未検証。Phase 1の
  アクションポーズシートで代替できない大きな可動域(伸ばす動作等)が
  実際に必要になった時点で、タイル生成方式での小規模スパイクから着手
- **極端姿勢(しゃがみ・ジャンプ等)**: 種類・枚数とも未確定。優先度は
  Phase 2の中でも低い
- 類似の既製ツール(Scenario「2D Animation Rigging Sheet Generator」、
  Viggle AI「JST-1」)の存在を確認済み。汎用ツールのためキャラ固有の
  識別要素の精度は未検証だが選択肢として認識しておく

### 組み立て方(アセンブリ)

- 生成したシート画像は**決定的スクリプト(AI不使用)でタイル位置ごとに
  切り出し**、マニフェストJSONの`frames`に登録する(`puppet/manifest.js`
  の座標形式を拡張)
- 描画順・ピボット・隠蔽先は`new2D3D`で確立済みの**分解境界の3原則**を
  そのまま踏襲する: 境界を切ってよいのは(a)動きがゼロになる関節の付け根、
  (b)他レイヤーの下に常時隠れる場所、の2種類のみ。一体の造形を貫通しない
- **二層構造を維持する**: 連続的な動き(呼吸・視線・揺れ・小角度首振り)は
  単一画像へのメッシュ変形/ソフトマスクで作り、離散状態(表情/viseme/
  手の形/アクション)は生成画像のクロスフェードで切り替える。**生成した
  複数のポーズ画像同士を頂点補間でつなぐことはしない**(ピクセル単位の
  連続性が保証されないため、モーフィングすると継ぎ目より先に中身が
  破綻するリスクが高い)
- QCゲート(`puppet/qc_gate.py`踏襲)は境界コンタクトシート方式のまま流用:
  生成直後に全パーツ極値のズームクロップで境界を確認してから採用する

### プロンプト設計

Gemini呼び出しは全カテゴリで以下の型に統一する(いずれも過去の検証
④⑤⑥⑧⑨⑭の教訓を反映):

1. **タイル配置・順序をプロンプト本文で明示する**: 「N×Mグリッドの
   コンタクトシートを1枚生成」「左から順に[状態A][状態B]…」のように、
   グリッド形状とタイルの並び順を文章で固定する。順序が守られないと
   後段の決定的スライス処理が壊れる
2. **識別要素は画像添付だけに頼らずテキストにも毎回明記する**:
   髪色・瞳の色・ヘッドセットの色/形・衣装配色を毎回プロンプト本文に
   書く。画像条件付けのみでは識別要素が安定しないという教訓(④⑤⑧、
   IP-Adapter系での検証だがGemini移行後も念のため踏襲)
3. **背景・線画スタイルを定型文言で固定する**: 「simple [color]
   background」「clean lineart, cel shading」等を全呼び出し共通の
   テンプレート文言にし、ネガティブ指定(装飾的な光の輪・不要な尻尾状
   パーツ等、過去に確認済みの不要要素)も定型化する
4. **参照画像との関係を明記する**: 「添付のマスター画像のキャラクターを
   ベースに、指定するタイル内容をそれぞれ生成する」のように、入力画像と
   出力の関係を文章で明示する
5. **変更してよい範囲/してはいけない範囲を分離する**: 「表情(目・口)
   以外は変更しない」のように対象を限定する一文を必ず入れる。この明記の
   有無が結果を左右すると確認済み(⑭)

### 残る不確実要素(次工程での検証事項)

- Gemini側でのタイル配置・順序遵守の精度は、表情シート以外(viseme・
  手の形・アクション)では未検証。タイル数やグリッド形状を変えた場合の
  再現性も未確認
- 手の形・アクションポーズシートは今回新規のカテゴリのため、最初の1回は
  少量(1シートのみ)で試してから本番投入する
- Phase 2(360度ビューリング・関節角度継ぎ足し)の同一キャンバス方式は
  まだ仮説段階。着手前に必ず小規模スパイクで検証すること

## パーツマニフェスト設計(たたき台)

`new2D3D`の既存パーツパペットエンジン(`puppet/manifest.js`)は座標ベース
の平面レイヤー(`x, y, w, h` + `anchors`)のみで、離散状態(タイル出自)を
持たない。Gemini移行後のマニフェストは、パーツごとに「タイルから切り出した
複数状態」を持てるよう拡張する。実装前の設計イメージ:

```json
{
  "size": [768, 1024],
  "template_version": "std-v1",
  "pivots_normalized": {
    "head": [0.50, 0.32],
    "shoulder_r": [0.62, 0.40],
    "shoulder_l": [0.38, 0.40],
    "hip": [0.50, 0.58]
  },
  "parts": {
    "face": {
      "motion": "discrete-crossfade",
      "states": {
        "neutral": { "x": 0, "y": 0, "w": 0, "h": 0 },
        "joy":     { "...": "..." },
        "aa":      { "...": "...", "group": "viseme" }
      },
      "source_sheet": "expr_sheet_v1.png"
    },
    "hand_r": {
      "motion": "discrete-crossfade",
      "states": { "open": { "...": "..." }, "fist": { "...": "..." } },
      "source_sheet": "hand_sheet_v1.png"
    },
    "hand_l": { "mirror_of": "hand_r" },
    "head": {
      "motion": "procedural-mesh",
      "range_deg": [-15, 15],
      "pivot": "head"
    },
    "ponytail": { "motion": "procedural-mesh", "pivot_source": "static_patch" }
  },
  "draw_order": ["base", "ponytail", "head", "face", "hand_l", "hand_r"]
}
```

`motion`が`discrete-crossfade`のパーツのみ生成シートを参照し、
`procedural-mesh`のパーツは生成不要(既存の`new2D3D`実装を踏襲)。この形式
は確定ではなく、実装着手時に見直す前提のたたき台。

## 次のアクション

1. Gemini APIキーの提供を待つ(このリポジトリのサンドボックスには未設定)
2. キー提供後、キャラクター基準画像(`new2D3D`の`characters/zero/`相当)を
   標準テンプレート(768×1024・L0ポーズ要件)に合わせて用意する
3. Phase 1の4カテゴリ(表情/viseme/手の形/アクション)を、シート1枚ずつ
   校正生成→検証のサイクルで漸進的に進める(`new2D3D`の検証プロトコルを
   踏襲: 全景判断禁止・反証駆動・差分画像の目視・証拠画像つき報告)。
   手の形・アクションは新規カテゴリのため特に慎重に
4. パーツパペットエンジン(WebGLビューア)本体は、Phase 1のマニフェスト
   設計が固まり次第このリポジトリで新規実装する(呼吸・視線・首振り・
   揺れのプロシージャル部分は生成物を待たずに先行実装可能)
5. Phase 2(360度ビューリング等)は、Phase 1が動いてから同一キャンバス
   タイル方式の小規模スパイクとして着手を検討する
