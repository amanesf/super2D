# PLAN.md — パーツ生成方式の現行方針(Gemini移行)

出典: [`amanesf/new2D3D`](https://github.com/amanesf/new2D3D) の
`SUPER_LIVE2D_V3_PLAN.md` §6.5(2026-07-21時点のまとめ)。本ドキュメントは
その内容をこのリポジトリでの実施計画として書き起こしたもの。実装コードは
まだ書いていない — Gemini APIキーが用意され、実際の生成呼び出しが許可され
た時点で着手する。

## 背景: なぜMeinaMix+ControlNet系を離れるのか

`new2D3D`では頭部360度ビューリング(Live2Dでは原理的に不可能な側面・背面
込みの全周表示)の実現手段として、SD1.5系(MeinaMix)+ControlNet /
IP-Adapter / img2imgチェーンを14パターン(①〜⑭)検証した。到達した結論:

- 単発の同一性(髪色・瞳・ヘッドセット・衣装)は、ステップ数とプロンプトの
  トークン管理を詰めれば実用域に達せられる(⑧)
- しかし**複数キーフレームを跨いだ一貫性**(同じ服の形・同じ瞳の色を毎回
  選ぶこと)は保証されない。同一設定で生成しても毎回ゼロから再解釈される
  ため、角度を並べたときに同一キャラの回転に見えない(⑨)
- 生のtxt2img(ControlNet/IP-Adapterなし)+同一seedの方が連続性はやや改善
  するが(⑪)、img2imgチェーンで実ピクセルを引き継ぐと連続性は完璧になる
  一方で回転が止まる(⑫⑬)— 「同一性」と「回転の自由度」がトレードオフ
  になり、SD1.5系の構成では両立しなかった

この構造的な限界を踏まえ、**パーツ生成の主軸をGemini 3.1 Flash Image
Preview(Nano Banana 2)に切り替える**。表情シート(1枚のキャンバスに複数
表情をタイル状に生成)で試したところ、髪色・瞳・ヘッドセット・衣装のいずれ
もMeinaMix系より大幅に安定して揃うことを確認済み(1枚のキャンバス内で生成
することで、モデルの注意機構が同一キャンバス内の情報を自然に参照し合い、
同一性が保たれやすいと推定)。

## 現在の方針

### パーツ分解設計

**角度軸(カメラ視点)と可動域軸(関節)を分離する。**

- 角度軸: 0〜330度(30度刻み)。左右対称のため片側7角度のみ生成し、残りは
  ミラーで作る
- 可動域軸: 関節を低可動部位と高可動部位に分ける
  - 低可動部位(頭・首・胴・腰): 各1〜2枚
  - 高可動部位(肩肘手首・股膝足首): 各3〜4枚、片側のみ生成しミラー。
    「1枚の絵を回転できる範囲を複数の絵で継ぎ足す」方式
  - 指: 骨ではなく手の形パッチの差し替え(開手/握り/指差し/マイク持ち等)
- 表情6種・viseme5種: 既存の低strength img2img差し替え方式を踏襲(変更なし)
- 極端姿勢(しゃがみ・ジャンプ等): 種類・枚数とも未確定

### 生成ボリューム見積もり(概算)

| 内訳 | 呼び出し回数 |
|---|---|
| 視点角度ベース | 7回 |
| 表情/viseme/手の形シート | 14回 |
| 極端姿勢 | 8回 |
| 関節可動パーツ(約30枚) | 約30回 |
| **合計** | **約60回** |

Gemini 3.1換算で**$4程度(¥600)**。最終アセット枚数は概ね120枚超。MeinaMix
路線の見積もり(数百枚・リトライ込みでより高コスト)より一桁小さい。

### テンプレート化・2キャラ目再利用への配慮

Geminiは**入力画像と同じサイズで出力する**性質があるため、標準化された
キャンバスサイズ・グリッドレイアウトを持つテンプレート入力画像を事前に
用意する。さらに標準プロポーション・標準関節ピボット位置をテンプレートに
含めておけば、リグ(ボーン位置・ワープメッシュ)をキャラ間で使い回せる
可能性がある(`PUPPET_ENGINE_DESIGN.md`の「2キャラ目コスト表」受け入れ
基準に対応する設計)。

### 未解決・要検証(次工程)

- 高可動パーツを角度切り替えでつないだときの継ぎ目・滑らかさ(未検証)
- 平面回転では奥行き方向の動き(手前に伸ばす等)を表現できない構造的限界
  がある(3Dとの本質的な差)
- 極端姿勢の具体的な種類・枚数
- 類似の既製ツール(Scenario「2D Animation Rigging Sheet Generator」、
  Viggle AI「JST-1」)が市場に存在することを確認済み。汎用ツールのため
  このキャラ固有の識別要素の作り込み精度は未検証だが、選択肢として認識
  しておく

## パーツマニフェスト設計(たたき台)

`new2D3D`の既存パーツパペットエンジン(`puppet/manifest.js`)は座標ベース
の平面レイヤー(`x, y, w, h` + `anchors`)のみで、角度軸・関節可動軸を持た
ない。Gemini移行後のマニフェストは、既存構造を拡張して角度・可動域の2軸を
表現できるようにする必要がある。実装前の設計イメージ:

```json
{
  "size": [670, 640],
  "parts": {
    "head": {
      "mobility": "low",
      "angles": [0, 30, 60, 90, 120, 150, 180],
      "mirror": true,
      "frames": {
        "0":   { "x": 0, "y": 0, "w": 0, "h": 0 },
        "30":  { "...": "..." }
      },
      "anchor": [0, 0]
    },
    "arm_r_upper": {
      "mobility": "high",
      "poses": ["rest", "raised", "forward", "back"],
      "mirror_of": null,
      "frames": { "rest": { "...": "..." } },
      "anchor": [0, 0]
    },
    "arm_l_upper": {
      "mobility": "high",
      "mirror_of": "arm_r_upper"
    }
  },
  "expressions": { "...": "既存 gen_expr.py / gen_viseme.py の差し替え方式を踏襲" },
  "draw_order": ["base", "arm_l_upper", "..."]
}
```

この形式は確定ではなく、実装着手時に見直す前提のたたき台。

## 次のアクション

1. Gemini APIキーの提供を待つ(このリポジトリのサンドボックスには未設定)
2. キー提供後、キャラクター基準画像(`new2D3D`の`characters/zero/`相当)を
   このリポジトリに持ち込み、テンプレート入力画像(角度ガイド・キャンバス
   規格)を設計する
3. 角度7枚→関節可動パーツ→表情/viseme/手の形の順に、校正1枚→小バッチ→
   検証のサイクルで漸進的に生成する(`new2D3D`の検証プロトコルを踏襲:
   全景判断禁止・反証駆動・差分画像の目視・証拠画像つき報告)
4. パーツパペットエンジン(WebGLビューア)本体は、マニフェスト設計が固まり
   次第このリポジトリで新規実装する
