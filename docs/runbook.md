# 実生成ランブック(Stage F)

これは**ユーザーが自分のGemini APIキーで実際にキャラクターを生成する**
ための手順書。コードではなく手順とチェックリストのみを扱う。
このリポジトリの開発セッションからは実APIを一切呼ばない
(CLAUDE.mdルール2)。この文書は`studio.html`(D1〜D7・E1で実装済み)を
使って実際に手を動かす人向け。

## 前提条件(すべて完了済み)

- レンダラ: WebGL(メッシュ+頂点スキニング)に確定済み(`PLAN.md`
  「C1決定」節)
- 振幅仕様と余白逆算値: 実装済みの値を「振幅仕様の実測値」節に記載
- Stage D(D1〜D7)・Stage E(E1)完了

## 用意するもの

- Gemini APIキー(Google AI Studioで発行。課金設定がされていないと
  429「limit: 0」で失敗するので注意。`js/gemini_api.js`がこのケースを
  検出してメッセージを出す)
- キャラクターの全身参照イラスト1枚(1024×1024相当を想定。異なる比率
  でも`js/normalize.js`が正規化する)

## 手順

### 1. 画像投入・APIキー設定

1. `studio.html`を開く(GitHub Pagesまたはローカル`python3 -m
   http.server`経由。`file://`では`fetch`が動かないタブがあるため
   簡易サーバーを推奨)
2. 「1. 画像投入・APIキー設定」タブで参照イラストをアップロード
3. Gemini APIキーを入力(localStorageにのみ保存される。未入力でも
   後続は手動フォールバック経路で進められる)

### 2. マスター生成

- (現時点ではタブの骨格のみ。実プロンプトでのマスター生成は本タスク
  範囲外のため、参照イラストをそのままマスターとして扱ってよい)

### 3. パーツ検出

1. 「3. パーツ検出」タブで`prompts/part_detection_v1.txt`の内容を確認
2. 経路A(APIで生成)/経路B(プロンプトをコピーして手動でGeminiに
   貼り付け)/経路C(生成済み結果を手動アップロード)のいずれかで
   パーツ構成JSONを取得する
3. 経路Aを使う場合、429エラーが出たらエラーメッセージを読む
   (レート制限か課金設定未完了かで対処が異なる)

### 4. パーツ別in-situ生成

1. `prompts/part_insitu_v1.txt`の`{PART_NAME}`/`{PART_DESC}`/
   `{MARGIN_NOTE}`/`{BACKGROUND_INSTRUCTION}`をパーツごとに埋めて実行
   (置換ロジックはStage D後続タスクで配線予定。現時点では手動で
   プレースホルダーを埋めてから経路B/Cで進める)
2. 生成結果は「4. パーツ別in-situ生成」タブのギャラリーに候補として
   追加される。複数候補から採用を選べる(ベストオブN)
3. 気になる候補には修正指示を出して会話継続編集ができる
   (**修正の連鎖は3回まで**。超えたら新規生成からやり直す)
4. 各パーツについて、次の「振幅仕様の実測値」節にある可動域で
   継ぎ目が破綻しないことを念頭に、余白(パーツ画像の透明マージン)が
   十分か確認する

### 5. モーション/継ぎ目QC

1. 「5. モーション/継ぎ目QC」タブで検査対象パーツを選び、実際の可動域
   (下表)でスライダーをスクラブする
2. 「自動往復ストレステスト開始」で連続的に確認する
3. 画面下の判定が「継ぎ目に隙間を検出」と出たら、そのパーツの余白
   (in-situプロンプトの`{MARGIN_NOTE}`)を広げて4に戻り再生成する

### 6. 圧縮・書き出し

1. `js/compress.js`(`S2D.compress.buildCharacterZip`)でsource(PNG)→
   public(WebP)を生成し、character.json+public+generation_logをZIPに
   まとめる(UIタブの配線は今回未着手。ブラウザのdevtoolsコンソールから
   `window.S2D.compress.buildCharacterZip(...)`を直接呼ぶ運用でも可)
2. 書き出したZIPを`viewer.html`にドラッグ&ドロップして表示確認する
3. 問題なければZIPを展開し、`characters/zero/`としてリポジトリに
   コミットする(mainへのpushはユーザーの明示指示があった時のみ)

## 各段階の合格基準

| 段階 | 合格基準 |
|---|---|
| 検出 | パーツ数・親子関係がマスター画像と矛盾しない |
| in-situ生成 | QCゲート(`js/qc_diff.js`)のdiff率が閾値以下(既定5%) |
| レジストレーション | `js/registration.js`補正後、マスターとの内容
  バウンディングボックスのズレが1px以内 |
| マッティング | `js/matting.js`のデュアル背景マッティングで位置ズレ
  (`positionMismatch`)が検出されない |
| 継ぎ目QC | 「5. モーション/継ぎ目QC」タブで実可動域を往復しても
  「継ぎ目に隙間を検出」にならない |
| 圧縮 | publicの合計サイズがsourceより有意に小さい(プレースホルダー
  検証では約29%減。実素材はグラデーション・陰影を含むためより大きく
  縮む見込み) |

## 振幅仕様の実測値(`scripts/make_placeholder_parts.js`実装値)

`PLAN.md`「振幅仕様が先、余白は逆算」節の目標値と、実装で使った
`blendMarginPx`(関節メッシュ変形の余白)の対応:

| パーツ | 動き | 振幅 | blendMarginPx |
|---|---|---|---|
| arm_lower_l/r(肘) | 待機の微揺れ | ±0.015rad(約0.86°) | 50px |
| arm_lower_l/r(肘) | 手振り時 | 概ね-0.95〜-0.25rad(約-54°〜-14°) | 50px |
| head_base(首) | 待機の首振り | ±0.02rad(約1.1°) | 60px |
| hair_front | 房内変形(遅延追従) | ±0.09rad(約5.2°) | 100px |
| hair_side_l/r | 房内変形(遅延追従、左右で位相・周波数をずらす) | ±0.05〜0.06rad
  (約2.9〜3.4°) | 25px/50px |
| torso | 呼吸スケール | ±3.5%(scaleX/Y) | (スケールのみ、角度なし) |

**重要な注意**: 「肘±40°」はPLAN.mdの目標値の例であり、実装(`updateWave`)の
実際の可動域は-54°〜-14°(中心-34°、振幅±20°)。継ぎ目QC画面
(D7)で確認したところ、この実運用範囲では継ぎ目の隙間は許容範囲内
(数px程度のアンチエイリアシング誤差のみ)だが、範囲外の角度(例: +30°、
手を振る方向と逆側)では明確な隙間が生じることを確認済み。**本番素材の
in-situ生成では、実際にキャラクターが取りうる角度範囲に絞って余白を
設計すること**(全方向±90°のような過大な余白は不要で、生成コストの
無駄になる)。
